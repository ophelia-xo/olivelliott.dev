  <action>
    Edit `next.config.ts`. Change the `options` block inside `createMDX` from:

    ```ts
    options: { remarkPlugins: [], rehypePlugins: [] },
    ```

    to:

    ```ts
    options: {
      // Strings (not function refs) are required for Turbopack compatibility in Next 16.
      // Source: https://nextjs.org/docs/app/guides/mdx#using-plugins-with-turbopack
      // remark-frontmatter strips YAML `---` blocks from the MDX AST so they don't render as body text.
      // gray-matter (in lib/content.ts) handles frontmatter parsing out-of-band via fs.
      remarkPlugins: ['remark-frontmatter'],
      rehypePlugins: [],
    },
    ```

    Do NOT:
    - Import `remarkFrontmatter` as a function and pass the reference (breaks Turbopack — confirmed in Next.js issue #84258).
    - Pass plugin options as a tuple `['remark-frontmatter', { type: 'yaml' }]` unless specifically required (default YAML parsing is what we want).
    - Add rehype plugins here — Shiki / `rehype-pretty-code` lands in Phase 3.

    Leave the rest of `next.config.ts` unchanged (`extension`, `pageExtensions`, `poweredByHeader`, `reactStrictMode`, the default export via `withMDX(nextConfig)`).

    After editing, smoke-gate by running `pnpm build`. It MUST succeed. This validates:
    1. `remark-frontmatter` string plugin resolves correctly under Turbopack.
    2. The new `mdx-components.tsx` stub satisfies the `@next/mdx` App Router contract.
    3. No regression to the Phase 1 baseline.
  </action>
